멀티 에이전트 tmux 운영 가이드
프로젝트: STAR TEAM — Robot Operations Manual

============================================================

[ 멀티 에이전트 동작 원리 ]

각 터미널에서 `claude`를 실행하면 독립적인 Claude Code 인스턴스가 됩니다.
에이전트들은 파일을 통해 소통합니다.
한 에이전트가 파일에 기록 → 다른 에이전트가 그 파일을 읽는 구조.

예:
  터미널1 (Lead)    → TASK-001.md 작성
  터미널3 (Backend) → TASK-001.md 읽고 작업 → 완료 기록 작성
  터미널2 (CI/CD)   → 완료 기록 읽고 커밋/배포


[ 6개 터미널 역할 배정 ]

에이전트가 7개, 터미널이 6개이므로
CEO + Lead Engineer는 T1에서 순차 처리합니다.

  T1 | CEO → Lead Engineer (Plan 확정 후 동일 터미널에서 Lead로 전환)
  T2 | UI/UX Designer
  T3 | Backend Engineer
  T4 | CI/CD Engineer
  T5 | QA
  T6 | Beta Tester


[ 진짜 병렬 실행의 두 가지 조건 ]

1. 권한 플래그
   claude --dangerously-skip-permissions
   이 플래그 없이는 파일 수정·bash 실행마다 승인을 기다립니다.

2. 완결된 프롬프트
   에이전트가 중간에 멈추지 않으려면 판단에 필요한 모든 정보가
   프롬프트 안에 있어야 합니다. 마지막에 반드시 포함:
   "모르는 것이 있으면 가장 단순한 방향으로 결정하고 기록에 남겨. 질문하지 마."


[ 이 프로젝트에서 처음부터 병렬로 돌릴 수 있는 작업 ]

  T1 Lead Engineer  → CYCLE-001.md 태스크 목록 작성
  T2 UI/UX          → index.html 경로 이슈 검사 및 수정
  T3 Backend        → nginx.conf + Dockerfile 작성
  T4 CI/CD          → git 정리 + .gitignore + deploy.sh 초안
  T5 QA             → 테스트 케이스 체크리스트 작성
  T6 Beta Tester    → 탐색 시나리오 준비 + python3 서버로 현재 상태 테스트

이 6개는 서로 기다릴 필요 없이 동시에 시작 가능합니다.

T1-A | CEO 역할 (먼저 실행)
------------------------------------------------------------
먼저 이 파일을 읽어: agents/ceo/SKILL.md, CLAUDE.md

너는 이 프로젝트의 CEO야.

현재 상황:
- public/index.html에 완성된 프로토타입이 있음 (6081줄 단일 HTML)
- CDN 의존성 사용 (marked.js, highlight.js, Pretendard 폰트 등)
- public/data/ 에 CSV/JSON 데이터 파일 존재
- src/assets/ 에 이미지/폰트 존재
- 빌드 시스템 없음

이번 사이클 목표를 정의해줘:
- 이 프로토타입을 그대로 배포 가능한 상태로 만드는 것
- 배포 형태: 이 서버(Linux)에서 Nginx로 서빙하거나 Docker 컨테이너로 패키징
- 성공 기준: 브라우저에서 모든 기능이 prototype과 동일하게 동작

CEO로서 이번 사이클의 목표, 성공 기준, 제약, 불포함 항목을
agents/ceo/PLAN-001.md 에 기록해줘.
기록 완료 후 나에게 알려줘.

------------------------------------------------------------
T1-B | Lead Engineer 역할 (CEO 완료 후 동일 터미널에서 전환)
------------------------------------------------------------
이제 Lead Engineer 역할로 전환해.
agents/lead_engineer/SKILL.md, agents/ceo/PLAN-001.md 를 읽어.

CEO의 Plan을 기반으로:
1. 이번 사이클 태스크를 분해해서 각 담당자에게 할당
2. agents/lead_engineer/CYCLE-001.md 에 전체 태스크 목록과 의존성 기록
3. 각 담당자별 태스크 파일을 agents/lead_engineer/tasks/ 디렉토리에 생성
   (TASK-001-backend.md, TASK-002-cicd.md, TASK-003-uiux.md, TASK-004-qa.md 등)

이 프로젝트는 정적 파일 배포이므로:
- Backend: Nginx 설정 또는 Docker 구성
- CI/CD: 배포 스크립트, git 정리
- UI/UX: 경로 이슈 검증 (상대경로, 이미지 로딩 등)
- QA: 배포 후 테스트 체크리스트
- Beta Tester: 탐색 테스트 시나리오

태스크 파일 작성 완료 후 나에게 알려줘.

T3 | Backend Engineer
------------------------------------------------------------
agents/backend_engineer/SKILL.md, CLAUDE.md 를 읽어.
그다음 agents/lead_engineer/tasks/TASK-001-backend.md 를 읽어.

너는 Backend Engineer야.

서빙할 수 있는 Nginx 설정 파일과 Docker 구성을 만들어줘.

고려할 것:

- HTTP로 먼저 동작하게 하고, HTTPS는 이후 단계
- 포트 80

출력 위치: 프로젝트 루트에 nginx.conf, Dockerfile, docker-compose.yml 생성
완료 후 TASK-001-backend.md 에 완료 기록 작성.

============================================================
실제 tmux 운영 절차
============================================================

1. 각 pane에서 시작:
   claude --dangerously-skip-permissions

2. tmux 동시 입력 (필요 시):
   tmux setw synchronize-panes on   ← 모든 pane 동시 입력
   tmux setw synchronize-panes off  ← 해제

3. 병렬 시작 순서:
   T1(CEO) 완료 → T1(Lead) 태스크 파일 생성
   → T2, T3, T4, T5, T6 동시 시작 가능
   → T4(CI/CD) 커밋은 T3(Backend) 완료 후
   → T5(QA) 검증은 T3+T4 완료 후
   → T6(Beta) 탐색은 서버 기동 후 즉시 가능
   → T1(Lead) 최종 Review 기록

4. 진행 상황 모니터링 (별도 터미널):
   watch -n 5 'ls -la agents/lead_engineer/tasks/'


============================================================
주의사항
============================================================

- 두 에이전트가 같은 파일을 동시에 수정하면 충돌 발생
- 작업 경계: index.html은 UI/UX만 / nginx.conf는 Backend만 / git은 CI/CD만
- 배포 관련 파일(Dockerfile 등)은 Backend가 생성, CI/CD가 커밋
- main 브랜치 직접 push 금지. feature/deploy → PR → merge 순서 준수
